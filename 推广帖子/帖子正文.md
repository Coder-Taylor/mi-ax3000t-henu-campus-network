#### Day 2 — 设备重连会断网

**【3.png】**
> WiFi 连接详情 — 连上就能用，不用每次登录

第二天发现一个问题：之前连过的手机重新连 WiFi 时，所有设备会短暂断网。

挂机游戏直接弹"网络异常"。

原因：部分设备（特别是老手机、单天线设备）连入时，会触发路由器芯片的 STA+AP 冲突。

修了 6 个地方：
- 5G 频宽从 80MHz 降到 40MHz（和上行对齐）
- 信道锁定 ch52（不让它自动跳）
- 写了个路由看门狗脚本，20 秒检测一次

**【4.png】**
> 路由器网口实拍 — 有线设备直接插，稳得一批

好了很多，但 20 秒还是太慢了。

#### Day 3 — 秒级恢复 + DNS 修复

**【5.png】**
> 流量消耗器 — 实测 102Mbps，下载速度 12.8MB/s

第三天，我决定把看门狗从 cron（20 秒一次）改成守护进程（每秒检测）。

踩了 9 个坑：
- `wifi reload` 太重了，会把所有设备踢掉，改成只重连指定信道
- `pidof` 匹配出问题，进程检测不准
- 守护进程挂了不会自动重启，要用 procd 管理
- dnsmasq 重启会影响 DNS，要单独处理

最后搞出来一个每秒检测的守护进程，断网 1-2 秒自动恢复。

**【6.png】**
> 压力测试结果 — 14 个站点 × 30 轮，全通过

同一天还修了校园内部网站的 DNS。

教务系统、图书馆、学工系统，校园网的 DNS 解析有问题，有的域名打不开。

研究了半天发现两个 DNS 服务器各有盲区：
- `111.6.174.198` 能解析教务，但学工打不开
- `114.114.114.114` 能解析学工，但教务返回内网 IP

解决方案：不同域名走不同 DNS，再加 hosts 文件写死。

#### 踩坑总结

| 问题 | 解决 |
|------|------|
| 设备重连全网断 | 频宽对齐 + 信道锁定 |
| cron 看门狗太慢 | 改成守护进程，每秒检测 |
| `wifi reload` 太重 | 改成只重连指定信道 |
| 校园网站打不开 | 不同域名走不同 DNS + hosts 写死 |
| Cloudflare 域名间歇失败 | hosts 写死 IP |
| 守护进程挂了不会重启 | 用 procd 管理 |
