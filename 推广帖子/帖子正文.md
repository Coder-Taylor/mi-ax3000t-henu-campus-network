### 正文

#### 项目背景

河大校园网用的是 Portal 认证，每次连 WiFi 要打开浏览器点登录，过几个小时认证过期还得重新来。四个人各买一张校园卡，每月套餐费加起来一百多块。

我想的是：能不能一台路由器搞定，四个人共用，自动认证，自动重连，还能跑满带宽？

设备选型：
- 路由器：小米 AX3000T（MT7981B + MT7976CN，WiFi6 三频），闲鱼全新 ¥125
- 固件：ImmortalWrt 24.10.5（OpenWrt 社区分支，开源免费）

技术方案：5G radio 同时做 STA（连校园网）和 AP（给宿舍设备用），写脚本自动完成 Portal 认证，cron + 守护进程保证断网自动恢复。

下面是完整的折腾过程。

#### Day 2 — 设备重连断网

**【3.png】**
> WiFi 连接详情 — 连上就能用

之前连过的手机重新连 WiFi 时，所有设备短暂断网。挂机游戏弹"网络异常"。

排查原因：部分设备（单天线、long preamble）关联时，触发了同芯片 STA+AP 冲突。5G radio 被迫 reset，全网断。

修了 6 个地方：

| 修复 | 效果 |
|------|------|
| 5G AP 频宽从 HE80 降到 HE40（和 STA 对齐） | 减少频宽不匹配触发 reset |
| 信道锁定 ch52 | 关闭 ACS 自动选择，消除周期性断连 |
| 路由看门狗 cron 20 秒检测 | 路由丢失后自动恢复 |
| 认证脚本升级 v2.0 | 认证 + 路由恢复一体化 |
| cron 优化 | 看门狗 20 秒，认证 5 分钟，分工明确 |
| init.d 启动脚本 | 开机自动启动 |

**【4.png】**
> 路由器网口实拍 — 有线设备直接插

好了很多，但 20 秒的恢复延迟还是太慢。

#### Day 3 — 秒级恢复 + DNS 修复

**【5.png】**
> 流量消耗器 — 实测 102Mbps，12.8MB/s

把 cron 看门狗改成守护进程，每秒检测。

踩了 9 个坑：

| 坑 | 解决 |
|----|------|
| `wifi reload` 太重，会踢掉所有设备 | 改成只重连指定信道 |
| `pidof` 匹配不准 | 用进程名 + PID 文件双重校验 |
| 守护进程挂了不会重启 | 用 procd 管理，自动 respawn |
| dnsmasq 重启影响 DNS | 仅在路由恢复时重启，加延迟保护 |
| procd 服务名冲突 | 重新定义服务标识 |
| 日志刷屏 | 加采样率控制，关键事件才记录 |
| 启动顺序问题 | START=99 保证最后启动 |
| 多进程并发 | 加锁机制 |
| 信号量残留 | 启动时清理旧信号量 |

最终：断网 1-2 秒自动恢复，比 cron 快 20 倍。

**【6.png】**
> 压力测试 — 14 站点 × 30 轮，全通过

同一天修了校园内部网站 DNS。

校园网 DHCP 下发两个 DNS 服务器，各有盲区：

| 域名 | 111.6.174.198 | 114.114.114.114 |
|------|:---:|:---:|
| jwgl.henu.edu.cn（教务） | ✅ 内网 IP | ❌ 外网 IP |
| xg.henu.edu.cn（学工） | ❌ | ✅ |
| zwyy.henu.edu.cn（资源） | ❌ 错误 IP | ✅ |
| lib.henu.edu.cn（图书馆） | ✅ | ✅ |

解决方案：
1. 不同域名走不同 DNS（UCI 域名转发）
2. `rebind-domain-ok=/henu.edu.cn/` 允许内网 IP
3. `/etc/hosts` 写死部分域名

#### 踩坑记录

| 问题 | 根因 | 解决 |
|------|------|------|
| 设备重连全网断 | STA+AP 频宽不匹配触发 reset | 频宽对齐 + 信道锁定 |
| cron 看门狗 20 秒太慢 | cron 最小粒度限制 | 改成守护进程每秒检测 |
| `wifi reload` 踢掉所有设备 | 全量重载无线 | 改成只重连指定信道 |
| 校园网站打不开 | 两个 DNS 各有盲区 | 域名级分流 + hosts 写死 |
| Cloudflare 域名间歇失败 | 校园 DNS 兼容性问题 | hosts 写死 IP |
| 守护进程挂了不会重启 | 无进程管理 | procd 管理 + 自动 respawn |
| dnsmasq 重启影响其他服务 | DNS 服务中断 | 加延迟保护 + 条件重启 |
