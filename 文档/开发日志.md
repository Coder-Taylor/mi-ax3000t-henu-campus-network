## Phase 0：前置知识——5 个核心概念

在理解我们做了什么之前，先建立这 5 个概念。它们贯穿了整个项目。

---

### 概念 1：SSH——我们如何"进入"路由器

```
你的电脑（Windows）                          路由器（Linux）
┌─────────────────┐                    ┌──────────────────┐
│  ssh root@      │  ───加密通道───▶   │  dropbear (SSH)   │
│  192.168.1.1    │                    │  在 192.168.1.1   │
│                 │  ◀───命令输出────  │  端口 22 监听      │
└─────────────────┘                    └──────────────────┘
```

- **SSH** = Secure Shell。它在你和路由器之间建立一条**加密的隧道**。
- 你敲的命令通过隧道发过去，路由器执行后把结果通过隧道传回来。
- OpenWrt 使用的 SSH 服务器叫 **dropbear**（比标准 OpenSSH 更轻量，适合嵌入式设备）。
- 连接命令：`ssh root@192.168.1.1`。`root` 是用户名，`192.168.1.1` 是路由器的 IP。
- **端口 22** 是 SSH 的默认端口。路由器上的 dropbear 一直在这个端口等待连接。

---

### 概念 2：OverlayFS——为什么重启不会丢失配置

这是理解 OpenWrt 系统的关键。普通 Linux 安装到一个硬盘分区，OpenWrt 则用了一种叫 **overlayfs** 的叠层文件系统：

```
                   你看到的完整文件系统 (/)
                  ┌──────────────────────────┐
                  │  /etc/config/network     │  ← 你修改的版本（来自 upper）
                  │  /etc/config/wireless    │
                  │  /usr/bin/my-script.sh   │
                  └──────────────────────────┘
                          ▲
            ┌─────────────┴─────────────┐
            │     /overlay (可写层)       │  ← ubifs，存在 NAND Flash 上
            │     你的所有修改在这         │     重启→保留 ✅
            │     大小：58MB              │
            └───────────────────────────┘
                          ▲  覆盖在上面
            ┌─────────────────────────────┐
            │     /rom (只读层)            │  ← squashfs，出厂固件
            │     原始系统文件              │     永远不被修改
            │     大小：7.8MB（已满）       │
            └─────────────────────────────┘
```

**关键理解**：
- `/rom` 是出厂固件，是只读的（squashfs 压缩文件系统）。
- `/overlay` 是空白的 writable 层（ubifs），存你的所有修改。
- 当你"修改" `/etc/config/network` 时，实际是：原始文件在 `/rom/etc/config/network`（只读），修改版本写到 `/overlay/upper/etc/config/network`。你看到的是上覆下。
- **恢复出厂设置 = 清空 `/overlay`**。重启不会丢，因为 overlay 在 NAND Flash 上。
- 这就是为什么我们说你的系统是"固化"的——不是临时的 initramfs。

### 概念 3：UCI——OpenWrt 的统一配置系统

传统 Linux 你要改网络就编辑 `/etc/network/interfaces`，改 WiFi 就编辑 `/etc/hostapd.conf`，改防火墙就改 `iptables` 规则……每个服务的配置文件格式都不同。

**OpenWrt 的解决方案：UCI（Unified Configuration Interface）**

```
    你（用户）
       │
       ▼
    UCI 命令（uci set / uci get / uci commit）
       │
       ▼
    /etc/config/ 下的统一配置文件
    ├── network   ← 网络接口、IP、DHCP
    ├── wireless  ← WiFi AP/STA、SSID、密码
    ├── firewall  ← 防火墙区域、NAT、转发规则
    ├── dhcp      ← DNS、DHCP 租约
    └── system    ← 主机名、时区
       │
       ▼
    各服务的初始化脚本自动把 UCI 配置翻译成服务自己的配置文件
```

**命令示例**：
```bash
uci set wireless.default_radio1.ssid='henu-student'   # 设置 WiFi SSID
uci get network.lan.ipaddr                              # 查看 LAN IP
uci commit wireless                                     # 保存修改
wifi reload                                             # 让 WiFi 服务重新读取配置
```

### 概念 4：NAT 和防火墙——路由器怎么"分享"网络

这是路由器的核心功能。假设路由器有两个接口：

```
  WAN 侧（校园网）                    LAN 侧（宿舍）
  IP: 10.36.82.65                    IP: 192.168.1.0/24
  ┌─────────────┐                   ┌─────────────┐
  │ phy1-sta0   │                   │ br-lan      │
  │ (5G STA)    │                   │ (LAN桥接)    │
  └──────┬──────┘                   └──────┬──────┘
         │                                 │
         └─────────── 路由器 ──────────────┘
                       │
                    转发 + NAT
```

**NAT（网络地址转换）做了什么**：

```
你手机 (192.168.1.249) → 发请求到 baidu.com (110.242.74.102)

手机的原始包：
  源IP: 192.168.1.249   →  目的IP: 110.242.74.102
  
路由器 NAT 后：
  源IP: 10.36.82.65     →  目的IP: 110.242.74.102
  (192.168.1.249 被替换成了路由器的 WAN IP)
  
baidu.com 回复：
  源IP: 110.242.74.102  →  目的IP: 10.36.82.65
  
路由器收到回复，反向 NAT：
  源IP: 110.242.74.102  →  目的IP: 192.168.1.249
  (路由器的 WAN IP 被替换回你手机的 LAN IP)
```

**没有 NAT 会怎样？** 手机的私有 IP（192.168.x.x）在公网上不可路由，baidu 无法回复。NAT 让所有宿舍设备的流量看起来都来自路由器自己的 WAN IP。

**OpenWrt 24.10 使用 nftables**（不是老旧的 iptables）。防火墙规则由 `fw4`（firewall4）从 UCI 配置自动生成。

---

### 概念 5：WiFi 的两种模式——AP vs STA

一个 WiFi 芯片（radio）只能工作在一种模式下……吗？实际上可以同时创建多个**虚拟接口（VIF）**，但功能有限制。

```
AP 模式（Access Point，接入点）     STA 模式（Station，客户端）
你手机连 WiFi 热点的模式          你手机连家里的 WiFi 路由器的模式
┌─────────────┐                    ┌─────────────┐
│  发射WiFi信号 │                    │  接收WiFi信号 │
│  让别人连     │                    │  去连别人     │
│  SSID: xxx   │                    │  连 SSID: xxx │
└─────────────┘                    └─────────────┘
   路由器做热点                          路由器做客户端
```

我们最后的状态：
- **radio0 (2.4G)**: AP 模式 → 发射 `可以和我在一起吗` 给宿舍用
- **radio1 (5G)**: STA 模式 → 连接 `henu-student` 作为上网出口

## Phase 1：路由器现状勘查（2026-06-04）

### 目标
搞清楚路由器当前刷了什么系统、处于什么状态、能不能永久保存配置。

### 操作与分析

#### 1.1 建立 SSH 连接

```bash
ssh root@192.168.1.1
```

> **小白学**：SSH 连接的三个要素——`用户名@IP地址`。`root` 是 Linux 的超级管理员账户，拥有最高权限。`192.168.1.1` 是路由器的 LAN 口 IP，是最常见的默认网关地址。我们通过网线连在路由器的 LAN4 口，所以能访问这个 IP。

#### 1.2 查看操作系统版本

```bash
cat /etc/os-release
```

输出：
```
NAME="ImmortalWrt"
VERSION="24.10.5"
OPENWRT_BOARD="mediatek/filogic"
OPENWRT_ARCH="aarch64_cortex-a53"
```

> **小白学**：`cat` 是 "concatenate" 的缩写，但实际作用就是"打印文件内容到屏幕"。`/etc/os-release` 是 Linux 标准的系统信息文件。我们看到：
> - **ImmortalWrt**：这是 OpenWrt 的一个中国社区维护的衍生版（对中文用户更友好，预装中文语言包和国内镜像源）。
> - **24.10.5**：版本号。24 是年份，10 是月份，5 是小版本。
> - **mediatek/filogic**：硬件平台。你的 AX3000T 使用联发科 MT7981B（Filogic 820）芯片。
> - **aarch64**：ARM 64 位架构，和手机 CPU 同类型。

#### 1.3 确认系统是否固化（最关键的问题）

```bash
mount          # 查看所有挂载点
df -h          # 查看磁盘使用
cat /proc/mtd  # 查看闪存分区表
```

> **小白学**：
> - `mount` 显示当前所有文件系统的挂载情况。Linux 中一切皆文件，磁盘分区要挂载到目录才能访问。
> - `df -h` 显示磁盘使用情况，`-h` 是 "human-readable"（比如显示 58M 而不是 589824）。
> - `/proc/mtd` 是内核暴露的闪存分区信息。`/proc` 是一个虚拟文件系统，里面的文件不占磁盘，是内核实时生成的。

**关键输出**：
```
/dev/root on /rom type squashfs (ro)     ← 只读固件层
/dev/ubi0_1 on /overlay type ubifs (rw)  ← 可写数据层 ← 这个存在！
overlayfs:/overlay on / type overlay     ← 叠层后的完整系统
```

分区表：
```
mtd8: 34MB  ubi_kernel    ← 内核 + 固件
mtd9: 78MB  ubi           ← 用户数据（可写 overlay）
```

> **小白学**：
> - **"ubi"是什么意思？** UBI = Unsorted Block Images，是专门为 NAND Flash 设计的存储管理层。NAND Flash 有坏块和磨损问题，UBI 帮你处理这些。
> - **为什么是 mtd8 和 mtd9 而不是 /dev/sda1？** 嵌入式设备没有硬盘。MTD = Memory Technology Device，是 Linux 对 NOR/NAND Flash 的抽象。
> - **判断结论**：因为看到了 `/dev/ubi0_1` 挂载为 `/overlay`（ubifs 格式，rw=可读写），说明系统已经**固化**。如果是在 initramfs 模式，就没有这个 ubifs overlay。

#### 1.4 查看当前网络和无线配置

```bash
cat /etc/config/network    # 网络接口配置
cat /etc/config/wireless   # 无线配置
cat /etc/config/firewall   # 防火墙配置
```

> **小白学**：`/etc/config/` 是 OpenWrt UCI 的配置目录。每个文件对应一个子系统。配置格式类似 INI 文件但有特定语法。

**关键发现**：
- LAN IP：`192.168.1.1`，桥接接口 `br-lan` 包含 lan2、lan3、lan4（lan1 被用作 WAN）
- WiFi：2.4G 和 5G 都是 AP 模式，SSID 为 `ImmortalWrt`，**无加密**（安全风险！）
- 系统无 root 密码（SSH 和 Web 都可无密码进入）

#### 1.5 确认硬件型号

```bash
ubus call system board
```

输出确认：`"model": "Xiaomi Mi Router AX3000T"`

> **小白学**：`ubus` 是 OpenWrt 的进程间通信总线。你可以把它理解成"系统信息的 API 接口"——各种服务把信息注册到 ubus，其他程序（包括 LuCI Web 界面）通过 ubus 查询。

### Phase 1 结论

| 问题 | 答案 |
|------|------|
| 系统是什么？ | ImmortalWrt 24.10.5（OpenWrt 衍生版） |
| 固件固化了吗？ | ✅ 已固化，overlay 在 NAND 上，重启不丢失 |
| 分区方案？ | 大分区版（mtd9 ubi 78MB，可用 58MB） |
| 有哪些安全问题？ | WiFi 无密码、root 无密码 |
| 能开始配置吗？ | 可以 |

## Phase 2：安全加固

### 目标
设置 root 密码、SSH 免密登录、WiFi 加密。

### 2.1 设置 root 密码——第一课：busybox 的坑

**第一次尝试**（在我们的电脑上执行）：
```bash
ssh root@192.168.1.1 "echo -e 'Czj20070312#\nCzj20070312#' | passwd root"
```

`passwd` 报告成功，但后来发现无法用新密码登录。

> **坑点分析**：问题的根源在于 **OpenWrt 使用 busybox**。
>
> **什么是 busybox？** 嵌入式 Linux 不用标准 GNU 工具（太大），而是用一个叫 busybox 的精简替代品。busybox 把所有常用命令（`ls`、`cat`、`echo`、`sh` 等）打包成一个程序。
>
> **busybox 的 `echo` 不支持 `-e`**！当你写 `echo -e 'xxx\nxxx'`，busybox 原样输出 `-e xxx\nxxx`，`\n` 没有被翻译成换行。这导致 `passwd` 把 `-e Czj20070312#\nCzj20070312#` 当成了密码（一整行，包含 `\n` 字面字符）。
>
> **教训**：嵌入式环境用 `printf` 代替 `echo -e`，因为 `printf` 的行为是标准化的。

**正确做法**（第二次）：
```bash
printf 'Czj20070312#\nCzj20070312#\n' | passwd root
```

`printf` 的 `\n` 行为在所有 shell 中都是一致的。

### 2.2 SSH 密钥免密登录——核心安全实践

**问题**：每次 SSH 输密码很麻烦，而且密码在网络传输（虽然是加密的）不如密钥安全。

**原理——非对称加密**：

```
你生成一对密钥：
  ├── 私钥 (ax3000t_rsa)    ← 永远留你电脑上，谁也不能给
  └── 公钥 (ax3000t_rsa.pub) ← 放到路由器上，公开没关系

连接时：
  路由器用公钥加密一个随机挑战 → 你的电脑用私钥解密 → 证明你有私钥 → 允许登录
  
  窃听者就算截获了挑战和响应，也算不出你的私钥（数学上不可行）
```

**操作步骤**：

```bash
# 1. 生成 ed25519 密钥对（你电脑上）
ssh-keygen -t ed25519 -f "C:/Users/HONOR/Desktop/AX3000T刷机校园网/.ssh/ax3000t_rsa"

# 2. 把公钥放到路由器上
#    公钥内容是一行文本，类似：
#    ssh-ed25519 AAAAC3NzaC1l... HONOR@LAPTOP-TAYLOR
#    把它追加到路由器的 /etc/dropbear/authorized_keys 文件中

# 3. 之后免密连接
ssh -i .ssh/ax3000t_rsa root@192.168.1.1
```

> **小白学**：
> - **为什么用 ed25519 而不是 RSA？** ed25519 是更新的算法，密钥更短（约 68 字符 vs 400+），安全性更高，性能更好。现代系统都支持。
> - **为什么是 `/etc/dropbear/` 而不是 `~/.ssh/`？** 因为路由器用的是 dropbear SSH 服务器，不是标准 OpenSSH。dropbear 的 authorized_keys 路径不同。
> - **SSH config 文件**：创建 `~/.ssh/config` 后，只需要打 `ssh ax3000t` 就能连接，不用每次输 IP 和密钥路径。

#### 途中遇到的 Windows 问题

在 Windows 上，SSH 不能直接从 stdin 读密码（因为 Windows 没有 `/dev/tty` 设备）。`sshpass` 工具也因此失效。

**解决方案**：`SSH_ASKPASS` 机制——写一个简单的脚本输出密码，通过环境变量告诉 SSH 用它来获取密码。这是 SSH 标准支持的密码自动化方式。

### 2.3 WiFi 加密配置

```bash
# UCI 命令修改无线配置
uci set wireless.default_radio0.ssid='可以和我在一起吗'
uci set wireless.default_radio0.encryption='psk2'
uci set wireless.default_radio0.key='好呀我也喜欢你!'
# ... 同样的命令对 radio1 执行 ...
uci commit wireless    # 保存到配置文件
wifi reload            # 让 hostapd 重新加载
```

> **小白学**：
> - **psk2 = WPA2-PSK**。PSK = Pre-Shared Key（预共享密钥），也就是你输的那个密码。WPA2 是目前最广泛使用的 WiFi 加密标准。
> - `uci set` 修改内存中的配置，`uci commit` 写入磁盘，然后通过服务的 `reload` 让配置生效。
> - WiFi SSID 在 `iw dev` 输出中显示为 `\xe5\x8f\xaf...` 是因为中文字符被转义成了十六进制（UTF-8 编码），实际显示是正常的。

<a name="phase3"></a>
## Phase 3：WiFi WAN 配置——让路由器无线连校园网

### 目标
把路由器的 5G radio 从"发射热点"改成"连接 henu-student"，作为上网出口（WAN）。

### 3.1 理解当前状态

配置前：
```
radio0 (2.4G): AP 模式 → SSID "可以和我在一起吗" → br-lan → 宿舍设备连
radio1 (5G):   AP 模式 → SSID "可以和我在一起吗" → br-lan → 宿舍设备连
WAN:           有线 lan1 → DHCP 获取 IP（插网线用）
```

我们的目标改造后：
```
radio0 (2.4G): AP 模式 → SSID "可以和我在一起吗" → br-lan → 宿舍设备连
radio1 (5G):   STA 模式 → 连接 henu-student → wwan → WAN → 外网
WAN:           无线 wwan（替代有线）
```

### 3.2 WiFi 扫描——找到 henu-student

```bash
iw dev phy0-ap0 scan | grep -E "SSID|signal|freq"
```

关键输出：
```
2.4G: henu-student  signal: -56 dBm  (信号好)
5G:   henu-student  signal: -78 dBm  (信号弱但够用)
```

> **小白学**：
> - **dBm 是什么？** 信号强度的单位，是负值。越接近 0 信号越好。-30 是极强，-50 是很好，-70 是及格，-80 以上不稳定。
> - **为什么选 5G 而不是 2.4G 连校园网？** 虽然 2.4G 信号更好，但 2.4G 穿墙能力强，留给宿舍设备更合适。5G 的 -78dBm 对于上网足够，且 5G 干扰少、速度快。
> - **`iw` 命令**：Linux 无线管理的底层工具（基于 nl80211 接口）。

**⚠️ 事故**：扫描后 SSH 短暂断开。MediaTek mt76 驱动在扫描时不太稳定。之后避免使用 `iw scan`。

### 3.3 修改无线配置

```bash
# 备份（好习惯！）
cp /etc/config/wireless /etc/config/wireless.bak

# radio1 改为 STA 客户端模式
uci set wireless.default_radio1.mode='sta'
uci set wireless.default_radio1.ssid='henu-student'
uci set wireless.default_radio1.encryption='none'   # 校园网无加密
uci delete wireless.default_radio1.key               # 删除WiFi密码
uci set wireless.default_radio1.network='wwan'        # 绑定到 wwan 网络
uci set wireless.radio1.channel='auto'               # STA模式自动选信道
uci commit wireless
```

> **小白学**：关键的几行：
> - `mode='sta'`：把无线接口从 AP（发射）改成 STA（接收/客户端）。
> - `encryption='none'`：henu-student 没有 WiFi 密码（开放的），它用 Portal Web 认证而不是 WiFi 加密。
> - `network='wwan'`：这是 UCI 的引用——让这个 WiFi 接口属于 `wwan` 这个逻辑网络。`wwan` 我们接下来创建。

### 3.4 创建 wwan 网络接口

```bash
uci set network.wwan='interface'
uci set network.wwan.proto='dhcp'
uci commit network
```

> **小白学**：`wwan` = Wireless WAN（无线广域网）。DHCP 协议让路由器从校园网的 DHCP 服务器自动获取 IP 地址、网关、DNS。

### 3.5 防火墙配置

```bash
uci add_list firewall.@zone[1].network='wwan'
uci commit firewall
```

> **小白学**：
> - `firewall.@zone[1]` 指防火墙配置中的第二个 zone（index 从 0 开始）。zone[0] 是 LAN，zone[1] 是 WAN。
> - `add_list` 向 WAN zone 的 network 列表添加 `wwan`，这样 wwan 接口的流量会被 NAT 和 WAN 防火墙规则覆盖。

### 3.6 应用并验证

```bash
wifi reload     # 让无线配置生效
```

`wifi reload` 做了什么？
1. 停止所有无线接口
2. 重读 `/etc/config/wireless`
3. 启动 radio1 的 STA 接口（去连 henu-student）
4. 启动 radio0 的 AP 接口（保持宿舍热点）

验证结果：
```
iw dev phy1-sta0 link

Connected to 06:74:9c:6a:71:9d (on phy1-sta0)
    SSID: henu-student
    signal: -49 dBm              ← 实际连接后信号比扫描时更好！
    rx bitrate: 400 MBit/s        ← 接收速率 400Mbps
```

然后获取 IP：
```bash
ifup wwan
ip addr show dev phy1-sta0
# 输出：inet 10.36.82.65/15
```

> **小白学**：`10.36.82.65/15` 中 `/15` 是子网掩码的 CIDR 表示法，等同于 `255.254.0.0`，意味着这个子网有约 13 万个可用 IP。

网络连通性测试：

<a name="phase4"></a>
## Phase 4：校园网认证脚本

### 目标
让路由器自动完成 Portal 认证的三步流程。

### 4.1 理解 HENU 校园网的认证机制

校园网使用了 **Portal 认证**（也叫 Captive Portal）。这是一种"先连 WiFi，再通过网页认证"的机制：

```
1. 你的设备连上 henu-student（开放WiFi，无密码）
2. DHCP 分配校园网内网 IP（如 10.36.82.65）
3. 此时可以访问校园内网（认证服务器、学校网站）
4. 但不能上外网（百度、谷歌、8.8.8.8）
5. 打开浏览器 → 被强制重定向到 Portal 页面 (172.29.35.36:6060)
6. 在页面上输学号密码 → 认证通过 → 外网通了
```

我们的脚本就是**模拟第5-6步**，用 HTTP 请求代替人工在网页上填写。

### 4.2 三步认证的 HTTP 细节

参考仓库：https://github.com/jry21223/HENU_autologin

```
第一步：验证账号
  POST http://172.29.35.27:8088/aaa-auth/api/v1/auth
  数据: campusCode + username + password + operatorSuffix
  期望: {"code":1}
  
第二步：二次校验
  POST http://172.29.35.27:8882/user/check-only
  数据: username + password + operatorSuffix
  期望: {"code":1}

第三步：门户最终认证
  GET http://172.29.35.36:6060/quickauth.do
  参数: userid + passwd + wlanuserip + wlanacname + wlanacIp + timestamp + uuid
  期望: {"message":"认证成功"}
```

> **小白学**：
> - **POST vs GET**：POST 把数据放在请求体里（`--data-raw` 或 `--post-data`），适合提交表单。GET 把参数放在 URL 的 `?` 后面，适合查询。
> - **172.29.x.x**：这是校园网内网 IP 段。只有连上 henu-student 才能访问这些服务器。
> - **Referer**：HTTP 头，告诉服务器"我是从哪个页面来的"。认证服务器校验这个头，需要假装来自 Portal 页面。
> - **Cookie**：第三步用了一个固定的 Cookie（`ABMS=362ee66b...`），可能是绕过某些校验。

### 4.3 curl 还是 wget——嵌入式环境的取舍

**第一个坑**：路由器没有预装 `curl`。

OpenWrt 的 `busybox` 自带了一个精简版 `wget`。它功能少但覆盖面窄。我们尝试 `opkg install curl`，但因为还没认证上不了外网，下载失败（鸡生蛋蛋生鸡）。

**解决方案**：用 busybox `wget` 重写脚本。

两者的语法对比：

| 功能 | curl | busybox wget |
|------|------|-------------|
| GET 请求 | `curl -s URL` | `wget -q -O - URL` |
| POST 请求 | `curl -X POST --data-raw "data"` | `wget --post-data="data"` |
| 自定义头 | `-H "Header: value"` | `--header="Header: value"` |
| Cookie | `-b "cookie"` | `--header="Cookie: cookie"` |
| 超时 | `--connect-timeout 10` | `-T 10` |

### 4.4 脚本核心结构解析

```bash
#!/bin/sh
# ↑ shebang——告诉系统用哪个解释器执行这个脚本

# 配置变量（改这里就能适配不同账号）
USERNAME="2510250328"
PASSWORD="386095"
OPERATOR_SUFFIX="@henuyd"    # 运营商后缀（henuyd=移动）
CAMPUS_CODE="07cdfd23373b17c6b337251c22b7ea57"

# 日志函数——所有输出带时间戳写入日志文件
log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" >> $LOG_FILE
}

# 网络检测——ping 通 8.8.8.8 说明已认证
check_network() {
    ping -c 2 -W 3 8.8.8.8 > /dev/null 2>&1
}
```

> **小白学**：
> - `#!/bin/sh` 不是注释！它是 Unix 的 shebang 机制，告诉内核"用 /bin/sh 来执行这个文件"。
> - `$(date '+%Y-%m-%d %H:%M:%S')` 是命令替换——先执行 `date` 命令，把输出嵌入字符串。
> - `> /dev/null 2>&1` 是把标准输出和错误输出都丢弃（只关心 ping 的退出码）。
> - `ping -c 2 -W 3`：`-c 2` 发 2 个包，`-W 3` 超时 3 秒。

### 4.5 测试结果

```
第一步响应: {"code":1,"msg":"发起登录","data":"2510250328@henuyd"}
第二步响应: {"code":1,"msg":"SUCCESS","data":null}
第三步响应: {"code":"0","rec":null,"message":"认证成功"...}

ping 8.8.8.8: 56ms ✅  外网通了！
```

> **小白学**：注意到第三步的 `"code":"0"` 不是错误——不同的 API 端点有不同的返回格式。关键看 `"message":"认证成功"`。

## Phase 5：定时重连 & NAT 修复

### 5.1 Cron 定时任务

```bash
echo "*/5 * * * * /etc/campus_network/auto_login.sh" >> /etc/crontabs/root
/etc/init.d/cron restart
```

> **小白学**：
> - **Cron** 是 Unix/Linux 的定时任务调度器。它每分钟醒来一次，检查有没有到时间的任务。
> - **crontab 格式**：`分 时 日 月 星期 命令`。`*/5 * * * *` 表示"每 5 分钟执行一次"（在所有小时、所有日期、所有月份、所有星期）。
> - OpenWrt 的 crontab 文件在 `/etc/crontabs/root`（root 用户的 cron 表）。
> - 脚本第一步就是 `check_network`，如果 ping 8.8.8.8 通了就直接退出（防止频繁认证），不通才走完整认证流程。这样设计很优雅。

### 5.2 NAT 修复——firewall4 的 fullcone 问题

**症状**：路由器自己能上网，但连在 WiFi 上的手机显示"无互联网"。

**诊断过程**：

```
1. 检查 NAT 规则：nft list chain inet fw4 srcnat_wan
   → 发现规则是 fullcone，没有传统 masquerade
   
2. 抓 conntrack：cat /proc/net/nf_conntrack
   → 发现从 br-lan (192.168.1.1) 出去的包没有被 NAT
   
3. 关闭 fullcone，改用标准 masquerade
```

> **什么是 fullcone NAT？** Fullcone（完全锥形）NAT 会为每个内网 IP:端口创建一个固定的外网映射。但我们的场景中，fullcone 似乎没有正确处理从 LAN 到 WAN 的新流量。

**修复**：
```bash
uci set firewall.@defaults[0].fullcone='0'
fw4 restart
```

`fw4 restart` 会重新生成 nftables 规则。现在 srcnat_wan 链变成了：
```
meta nfproto ipv4 masquerade
```

> **小白学**：`masquerade` 是 SNAT 的一种特殊形式——自动使用出口接口的 IP 作为源地址。比手动指定 SNAT IP 更灵活，适合 DHCP 获取 IP 的场景。

**验证**：PC（通过以太网连路由器）可以 ping 8.8.8.8，说明 NAT 生效了。手机显示"无互联网"只是图标问题，实际能上网。

> **关于手机"无互联网"图标**：Android/iOS 通过访问特定的 URL（如 `http://connectivitycheck.gstatic.com/generate_204`）来判断 WiFi 是否有互联网连接。如果这个请求被中间设备劫持或超时，手机会错误显示"无互联网"。**以浏览器实测为准！**

<a name="pitfalls"></a>
## 踩坑记录 & 故障排查技巧

### 坑 1：busybox echo 的 -e 陷阱

**现象**：用 `echo -e 'xxx\nxxx' | passwd` 设密码，设完登录不了。

**根因**：busybox 的 `echo` 不支持 `-e` 选项。`\n` 被当字面字符处理。

**解决**：永远用 `printf` 代替 `echo -e`。

**通用技巧**：在嵌入式环境，不确定命令行为时，加一个小测试：
```bash
echo -e "test\n" | od -c    # 看实际输出了什么
```

### 坑 2：Windows /dev/tty 缺失

**现象**：`sshpass` 在 Windows 上无法工作。

**根因**：`sshpass` 通过 `/dev/tty` 注入密码，Windows 没有这个设备。

**解决**：用 `SSH_ASKPASS` + `SSH_ASKPASS_REQUIRE=force` 环境变量方式。

**通用技巧**：Windows 做 SSH 自动化，最佳实践是用 SSH 密钥（无密码）而非密码认证。

### 坑 3：mt76 WiFi 驱动不稳定

**现象**：`iw scan` 后 SSH 连接断开。

**根因**：MediaTek mt76 驱动在扫描时会短暂重置无线芯片，如果 SSH 流量恰好在那一刻经过可能与无线相关的路径，会超时。

**解决**：减少不必要的扫描操作，扫描后等待稳定。

### 坑 4：opkg 装不了包（认证前无外网）

**现象**：`opkg install curl` 失败，因为路由器还没认证不能上外网。

**根因**：校园网的 Portal 认证机制——认证前只能访问内网。opkg 源在外网。

**解决**：用已有的工具（busybox wget）代替需下载的工具（curl）。

**通用技巧**：先检查有什么 → 再改造 → 最后才尝试安装新东西。

### 坑 5：firewall4 fullcone NAT 不生效

**现象**：路由器能上网，LAN 客户端不能。

**根因**：ImmortalWrt 默认开启的 fullcone NAT 在某些网络环境下不能正确处理来自私有子网的流量。标准 masquerade 更可靠。

**诊断方法**：
```bash
# 1. 检查 NAT 规则
nft list chain inet fw4 srcnat_wan

# 2. 查看连接跟踪（看包是否被 NAT）
cat /proc/net/nf_conntrack | grep 8.8.8.8

# 3. 从路由器本机测试（绕过 NAT）
ping 8.8.8.8          # 路由器自己用 phy1-sta0 的 IP 出去
ping -I br-lan 8.8.8.8  # 用 LAN IP 出去（需要 NAT）
```

## 扩展篇 1：路由器硬件原理深度解析

> 这一章讲"路由器到底是什么"。理解了硬件，你就能判断这台设备能做什么、不能做什么。

### 1.1 路由器 = 一台微型电脑

你的 AX3000T 拆开外壳后，里面是这些东西：

```
┌─────────────────────────────────────────────────────────────┐
│                                                               │
│   ┌─────────────┐   ┌──────────────────────────────────┐    │
│   │  NAND Flash  │   │  MediaTek MT7981B (Filogic 820) │    │
│   │  128MB       │   │  ┌────────────────────────────┐ │    │
│   │  (相当于硬盘) │◄──┤  │  ARM Cortex-A53 ×2核心      │ │    │
│   │              │   │  │  主频: 1.3GHz                │ │    │
│   └─────────────┘   │  │  相当于: 手机的CPU性能         │ │    │
│                     │  └────────────────────────────┘ │    │
│   ┌─────────────┐   │                                  │    │
│   │  DDR4 RAM   │   │  ┌────────────────────────────┐ │    │
│   │  256MB       │◄──┤  │  网络加速引擎 (PPE)          │ │    │
│   │  (相当于内存)│   │  │  硬件NAT, 不消耗CPU           │ │    │
│   │              │   │  └────────────────────────────┘ │    │
│   └─────────────┘   │                                  │    │
│                     │  ┌────────────────────────────┐ │    │
│   ┌─────────────┐   │  │  MT7976C 无线芯片            │ │    │
│   │  4×千兆网口  │◄──┤  │  2.4G + 5G 双频              │ │    │
│   │  (交换机芯片) │   │  │  2×2 MIMO (两根天线)        │ │    │
│   │              │   │  └────────────────────────────┘ │    │
│   └─────────────┘   └──────────────────────────────────┘    │
│                                                               │
│   功耗: ~5-12W  (一年电费 ≈ 30-70元)                          │
│   发热: 被动散热 (无需风扇)                                    │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 每个部件详解

#### SoC（片上系统）—— MT7981B "Filogic 820"

这是路由器的"大脑"。现代路由器 SoC 包含不止 CPU：

```
MT7981B 内部结构：
┌──────────────────────────────────────┐
│  双核 ARM Cortex-A53 @ 1.3GHz        │  ← 通用的CPU核心，跑Linux
│                                       │
│  PPE (Packet Processing Engine)      │  ← 硬件加速数据包转发
│    - 硬件NAT (不消耗CPU就能转发流量)   │     没有这个，路由器跑不满千兆
│    - QoS 硬件队列                     │
│                                       │
│  内置 2.5G 以太网 MAC                 │  ← 有线网络接口
│                                       │
│  PCIe 2.0 控制器                      │  ← 连接 WiFi 芯片
│                                       │
│  USB 3.0/2.0 控制器 (部分型号)        │  ← 可以接U盘/硬盘
│                                       │
│  加密加速引擎 (EIP197)                │  ← 硬件加解密，VPN加速
└──────────────────────────────────────┘
```

> **小白学**：
> - **为什么路由器 CPU 不如手机 CPU 快，却能同时处理几十个设备？** 因为有 PPE 硬件加速。普通数据包转发不经过 CPU，而是由专门的硬件引擎处理。CPU 只处理"特殊"流量（如 VPN、QoS、防火墙规则）。
> - **Cortex-A53** 是 ARM 在 2012 年发布的 64 位处理器核心，至今广泛用于中低端手机、IoT 设备、路由器。性能大约相当于骁龙 625 的水平。

#### 内存 (RAM) —— 256MB DDR4

```
路由器的内存分布 (典型状态):
┌─────────────────────────┐
│  Linux 内核:      ~30MB │
│  各服务进程:       ~20MB │
│  WiFi 驱动缓存:   ~10MB │
│  磁盘缓存 (buffer):~40MB │
│  NAND 文件系统缓存:~30MB│
│  空闲:           ~126MB │
└─────────────────────────┘
总大小: 256MB
```

> **为什么只有 256MB 却够用？** 路由器不需要图形界面（那才是最吃内存的）。纯命令行 Linux 在 128MB 内存下就能流畅运行。你的 AX3000T 的 256MB 对于 OpenWrt 来说已经算"豪华"配置。

#### 存储 (Flash) —— 128MB SPI NAND

```
128MB NAND Flash 布局:
┌──────────────────────────────┐
│  BL2        (1MB)  ← 芯片启动代码      │
│  Nvram/Bdata (512KB) ← U-Boot 环境变量 │
│  Factory    (2MB) ← WiFi 校准数据       │  ← 千万不能擦掉！
│  FIP        (2MB) ← U-Boot + ATF       │
│  ubi_kernel (34MB) ← Linux内核+固件    │
│  ubi        (78MB) ← 用户数据(我们用的) │
│  crash等    (2.5MB) ← 调试分区          │
└──────────────────────────────┘
```

> **关键理解**：Factory 分区存储了每台路由器唯一的 WiFi 校准参数（发射功率、频率偏移等）。刷机时绝对不能擦除这个分区，否则 WiFi 信号质量会严重下降，且无法恢复。

#### 无线芯片 —— MT7976C

```
2.4G 频段 (radio0):                  5G 频段 (radio1):
  频率: 2.412 - 2.472 GHz              频率: 5.180 - 5.825 GHz
  信道: 1-13 (我们用的信道1)            信道: 36-165 (我们连在 52)
  天线: 2根 (2×2 MIMO)                 天线: 2根 (2×2 MIMO)
  最大速率: 574 Mbps (WiFi 6)          最大速率: 2402 Mbps (WiFi 6)
  穿墙能力: ★★★★☆                     穿墙能力: ★★★☆☆
  干扰情况: 较多 (蓝牙、微波炉...)      干扰情况: 较少
```

### 1.3 数据包在路由器内部的旅程

当一个手机通过 WiFi 访问 `baidu.com` 时，数据包在路由器内部的完整路径：

```
                         路由器内部

  手机                     │                    外网
  (192.168.1.249)         │                   (baidu.com)
      │                    │                      │
  ① WiFi帧                │                   ⑧ 以太网帧
      ↓                    │                      ↑
  phy0-ap0 (2.4G AP)      │              phy1-sta0 (5G STA)
      ↓                    │                      ↑
  ② 802.11→以太网转换      │          ⑦ 以太网→802.11 封装
      ↓                    │                      ↑
    br-lan (软件桥)         │               nftables NAT
      ↓                    │             ⑥ 替换源IP: 192.168.1.249
  ③ 桥转发                 │                → 10.36.82.65 (SNAT)
      ↓                    │                      ↑
  nftables INPUT           │              nftables FORWARD
  ④ 允许 (LAN→WAN)         │          ⑤ 允许 (accept_to_wan)
      ↓                    │                      ↑
  ┌───┴─────────────────┴──┐
  │   Linux 网络栈 (IP路由)  │
  │  查路由表: 8.8.8.8的下一跳 │
  │  → via 10.36.0.1 dev    │
  │    phy1-sta0             │
  └──────────────────────────┘
```

> **每一步都在做什么**：
> 1. 手机通过 WiFi 发射 802.11 格式的无线帧
> 2. phy0-ap0 接收 WiFi 信号，去除 802.11 头部，变成标准以太网帧
> 3. br-lan（软件桥）把帧从 phy0-ap0 转发到 Linux 网络栈
> 4. iptables/nftables 的 FORWARD 链检查 LAN→WAN 是否允许
> 5. 允许通过
> 6. NAT（SNAT）把源 IP 从 `192.168.1.249` 替换为 `10.36.82.65`
> 7. 封装成 802.11 帧（因为出口是 WiFi STA）
> 8. 通过 phy1-sta0 发射到空中，去 henu-student AP

**这个路径是最慢的"软件路径"。** 如果 WAN 口是有线（普通以太网），PPE 硬件加速可以直接在硬件层面完成 ②-⑦，数据包根本不经 CPU。这就是为什么有线 NAT 能跑满千兆而 WiFi 不能。

## 扩展篇 3：路由器的启动过程

> 理解启动过程对调试和救砖非常重要。

### 按下电源键之后……

```
    通电
      │
      ▼
  ┌──────────┐
  │  BL2      │  ← Boot Loader Stage 2
  │  (ROM里)  │     芯片出厂烧死的，无法修改
  │           │     初始化 DDR 内存
  │           │     加载 FIP 到内存
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  FIP      │  ← Firmware Image Package
  │  (Flash里)│     包含: U-Boot + ATF (ARM Trusted Firmware)
  │           │     ATF 配置安全环境 (TrustZone)
  │           │     U-Boot 初始化外设、读环境变量
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  U-Boot   │  ← 开源引导加载程序
  │           │     检查环境变量 (从哪里启动)
  │           │     从 ubi_kernel 分区加载 Linux 内核
  │           │     传递内核参数 (cmdline)
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  Linux    │  ← 内核启动
  │  Kernel   │     初始化驱动 (WiFi, 网卡, NAND...)
  │           │     挂载 squashfs 到 /rom
  │           │     挂载 ubifs 到 /overlay
  │           │     组合 overlayfs 为根文件系统 /
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  init     │  ← 第一个用户态进程 (PID 1)
  │  (procd)  │     读取 /etc/inittab
  │           │     启动各服务 (network, firewall, dnsmasq...)
  │           │     LuCI Web 界面启动
  └────┬─────┘
       │
       ▼
   🟢 正常运行
```

### 什么是"砖"和怎么救

```
砖的状态:
  - "软砖" (Soft Brick): 系统启动失败但 U-Boot 正常
    → 通过 U-Boot 的 TFTP/HTTP 恢复模式重刷固件
    
  - "硬砖" (Hard Brick): BL2 或 FIP 损坏
    → 需要用 TTL 串口 + 编程器重写 Flash (非常麻烦)
```

**AX3000T 的 U-Boot 恢复模式**：按住 Reset 键再通电，U-Boot 进入恢复模式，可以通过网线把固件 TFTP 刷入。这是最常用的救砖方式。

> **小白保命法则**：
> 1. 千万不要动 mtd0-mtd4（BL2, Nvram, Bdata, Factory, FIP）
> 2. 刷机只操作 mtd8 和 mtd9（ubi_kernel 和 ubi）
> 3. 备份 Factory 分区——丢了 WiFi 就废了
> 4. 只要 BL2+FIP 完好，路由器就一定能救回来

## Phase 6：网络延迟监控工具（2026-06-04）

### 目标
开发一套基于 Windows 内置 curl.exe 的校园网延迟监控工具，无需安装任何第三方软件。

### 6.1 需求分析

用户需要：
1. 实时监控多个站点的 TCP/SSL 延迟
2. 可视化展示（折线图、柱状图、颜色分级）
3. 压力测试功能（多轮采样）
4. 前后对比功能（对比两次测试结果）
5. 双击启动，不需要手动开 PowerShell

### 6.2 技术方案

**测量原理**：
```
curl -s -m 5 -w "%{time_connect} %{time_appconnect}" -o NUL URL

返回两个值（秒）：
  time_connect     → TCP 建联耗时
  time_appconnect  → SSL 握手耗时（HTTP 站点无此值）

乘以 1000 转换为毫秒
```

**监控站点（14个）**：
- 内网网关：192.168.1.1（仅 TCP）
- 国内站点：百度、B站、网易、京东、QQ、淘宝、微博、知乎、新浪
- 国际站点：Cloudflare、GitHub、微软、Apple

### 6.3 开发过程与踩坑

#### 坑 1：PowerShell Unicode 编码问题

**现象**：脚本中有 `═`（Unicode 双横线）和 `─`（Unicode 单横线）等特殊字符，PowerShell 5.1 解析器报错 `TerminatorExpectedAtEndOfString`。

**根因**：Windows PowerShell 5.1 对 UTF-8 无 BOM 文件中的多字节 Unicode 字符解析有 bug。

**解决**：
1. 把所有 Unicode 制表符替换为 ASCII 字符（`═` → `=`，`─` → `-`）
2. 用 `[System.Text.UTF8Encoding(true)]` 给 PS1 文件加 UTF-8 BOM 头

```powershell
$content = [System.IO.File]::ReadAllText($path, [System.Text.Encoding]::UTF8)
$utf8Bom = New-Object System.Text.UTF8Encoding($true)
[System.IO.File]::WriteAllText($path, $content, $utf8Bom)
```

#### 坑 2：光标定位刷新显示乱码

**现象**：用 `[Console]::SetCursorPosition(0, $line)` 做局部刷新，输出严重错位，只显示片段如 `0%)`, `(0%)`。

**根因**：`SetCursorPosition` 在某些终端环境下（特别是通过 Claude Code 的 bash 工具调用时）不可靠。缓冲区输出 `StringBuilder` + `Write-Host -NoNewline` 的组合在非交互式管道中表现异常。

**解决**：放弃光标定位方案，改用 `Clear-Host` 全屏刷新。虽然有轻微闪烁，但显示稳定可靠。

#### 坑 3：Batch 文件中文编码闪退

**现象**：双击 `启动监控.bat` 立即闪退，报错 `'cho' 不是内部或外部命令`。

**根因**：Batch 文件用 UTF-8 编码保存，但 cmd.exe 期望 ANSI（系统默认编码）。UTF-8 的多字节中文字符被 cmd.exe 错误解析，导致命令碎片化。

**解决**：
1. 用 PowerShell 的 `[System.Text.Encoding]::Default`（ANSI）重新写入 batch 文件
2. 菜单文字改用英文避免编码问题
3. 添加 `cd /d "%~dp0"` 确保工作目录正确

```powershell
[System.IO.File]::WriteAllText($path, $content, [System.Text.Encoding]::Default)
```

#### 坑 4：中文文件名在 Bash 中不可见

**现象**：在 Git Bash 中 `ls` 看不到中文名的 PS1 文件。

**根因**：Git Bash 的终端编码默认不是 UTF-8。

**解决**：文件名保持英文或用简单的中文，不影响 Windows 资源管理器中的正常使用。

### 6.4 最终文件结构

```
监控工具/
├── start.bat          ← ANSI 编码，双击启动菜单
├── Monitor.ps1        ← 实时监控（0.1秒刷新，XY折线图，网速，流量）
├── StressTest.ps1     ← 压力测试（站点可选×轮数可选）
├── Compare.ps1        ← 对比两次测试结果
└── 测试记录/           ← 自动保存的 JSON 数据
```

### 6.5 功能清单（v7.x）

**Monitor.ps1 实时监控**：
- 9 个站点（1 网关 + 8 外网），可选择部分或全部
- 并行测量（Start-Job），curl 超时 2 秒
- 实时网卡吞吐量（Get-NetAdapterStatistics），0.1 秒刷新
- 带宽容量测试（Cloudflare/腾讯/阿里 CDN），每 30 秒一次
- 累计流量统计（从监控开始计算）
- XY 坐标折线图（Y 轴用 P5-P95 百分位缩放，忽略尖峰）
- 颜色分级：绿(<50ms) 黄(50-150ms) 红(>150ms/FAIL)
- 结果自动保存为 JSON

**StressTest.ps1 压力测试**：
- 站点选择菜单（1,3,5 或 1-5 或 all）
- 轮数选择（10/30/60/100）
- TCP+SSL 延迟统计（Avg/Min/Max/P50/P95/P99/Jitter）
- 柱状图、走势图、稳定性排名、网络质量评估

**Compare.ps1 对比工具**：
- 列出所有测试记录
- 选择两次测试对比
- 显示每个站点的延迟变化趋势

### 6.5 关键技术点

**统计指标**：
- Avg（平均值）、Min（最小值）、Max（最大值）
- P50（中位数）、P95（第95百分位）、P99（第99百分位）
- Jitter（抖动/标准差）— 衡量稳定性

**颜色分级**：
- 绿色：TCP < 50ms（正常）
- 黄色：TCP 50-150ms（偏慢）
- 红色：TCP > 150ms（很慢）

**折线图实现**：
```
画布: 8行 × 50列 字符矩阵
数据: 最近30个采样点
映射: 值 → 行号（归一化到 0-7）
填充: 数据点用 '*'，下方用 '.'
```

**结果保存**：每次测试结束自动保存为 JSON 文件到 `测试记录/` 目录，文件名含时间戳，支持后续对比。

### 6.6 教训总结

| 教训 | 说明 |
|------|------|
| PowerShell 5.1 的 Unicode 支持很差 | 中文路径、特殊字符都可能出问题，必须加 BOM |
| Batch 文件必须用 ANSI 编码 | UTF-8 会导致 cmd.exe 解析错误 |
| `Clear-Host` 比 `SetCursorPosition` 可靠 | 虽然有闪烁，但不会出错 |
| 嵌入式脚本避免复杂终端操作 | 简单的 Write-Host 输出最稳定 |
| 给用户留"退出后看结果"的路径 | Ctrl+C 后的 finally 块是关键 |
| 并行测量用 Start-Job | 顺序测量 N 个站点 = N×超时时间，并行 = 1×超时时间 |
| 网卡统计比下载文件更实时 | Get-NetAdapterStatistics 每 0.1 秒可读，不消耗带宽 |
| 带宽容量需要单独测 | 网卡统计只显示当前使用量，不是最大容量 |
| 百分位缩放忽略尖峰 | P5-P95 范围做 Y 轴，避免一个尖峰把图表压扁 |
| 国际站点从校园网可能超时 | curl -m 2 秒快速失败，不影响其他站点刷新 |

#### 第二步：发现关键配置——5G 芯片双模运行

```
radio1 (5G MT7976C) — 同一块物理芯片
├── phy1-sta0: STA 客户端 → 连接 henu-student (WAN出口)  频道52, 40MHz
└── phy1-ap0: AP 热点     → 广播 "可以和我在一起吗-5G"    频道52, 80MHz ⚠️
```

> **小白学**：这相当于让手机同时开热点和连 WiFi——同一个 WiFi 芯片要同时"接收"（连校园网）和"发射"（本地 5G 热点）。芯片只能通过时分复用来切换，就像一个人同时做两件事，必然会互相干扰。

**两个关键问题**：
1. STA 和 AP 共用同一频道（52）但**频宽不匹配**（STA 40MHz vs AP 80MHz）
2. 当 AP 侧有新设备连接（auth → assoc → WPA 握手），芯片需要短暂切换到 AP 模式处理

#### 第三步：日志铁证

```bash
logread | grep -E 'hostapd|odhcpd|route|disconnect'
```

发现关键日志：

```
22:57:32  设备 ea:d3:9a... 连接到 phy1-ap0 (5G AP)
22:57:32  ⚠️ odhcpd: No default route present, setting ra_lifetime to 0!
22:57:41  设备 40:c7:3c... 连接到 phy0-ap0 (2.4G AP)
22:57:41  ⚠️ odhcpd: No default route present, setting ra_lifetime to 0!
```

> **小白学**：`odhcpd` 是 OpenWrt 的 DHCPv6 + Router Advertisement 守护进程。它的 "No default route present" 错误意味着：**路由器自己都不知道怎么上外网了**——默认路由消失了！RA lifetime 设为 0 意味着告诉所有 IPv6 客户端"别用我上网"。

dmesg 内核日志更进一步揭示了时序：

```
[21.7s] phy1-ap0: entered forwarding state     ← 5G AP 上线
[21.9s] phy1-ap0: left promiscuous mode        ← AP 被软件下线！
[22.0s] br-lan: port 5(phy1-ap0) entered disabled state
[24.0s] phy1-sta0: authenticate with henu-student  ← STA 开始认证
[24.1s] phy1-sta0: associated
[24.2s] phy1-ap0: entered forwarding state     ← AP 恢复
```

**问题链路完全清晰了**：

```
新设备连接 5G AP
  → hostapd 处理关联/WPA握手
    → mt76 驱动暂停 phy1-ap0 桥接端口
      → br-lan 拓扑变化
        → WiFi 芯片模式切换（80MHz ↔ 40MHz）
          → phy1-sta0 (WAN) 短暂断连
            → 默认路由丢失
              → odhcpd 报警
                → 所有设备断网 💀
```

> **小白学**：整个过程可能只持续 0.5-2 秒，但 TCP 连接（HTTP、游戏、视频通话）对这种瞬时中断非常敏感。连接一断就需要重新建立，用户感知就是"网断了"。

#### 第四步：为什么"旧设备"更容易触发？

| 旧设备特征 | 导致的问题 |
|-----------|-----------|
| 仅支持 802.11n (WiFi 4) 或更老 | 关联帧交换更多，WPA 握手更慢 |
| 不支持 WMM Power Save | 芯片无法高效时分复用 |
| 慢速 CPU 处理关联 | 整个关联过程耗时更长（200-500ms vs 50ms） |
| 可能频繁漫游（2.4G↔5G） | 反复触发桥端口状态变化 |
| 不支持 802.11ax (WiFi 6) | 无法利用 OFDMA 减少信道占用 |

现代 WiFi 6 设备关联只需 ~50ms，不足以触发路由丢失。但旧设备可能需要 200-500ms，这个时间窗口足够让 STA 侧断连。

### 7.2 根因总结

| 排名 | 原因 | 可能性 | 证据 |
|------|------|--------|------|
| 🥇 | **5G 芯片 AP+STA 频宽不匹配** | 极高 | dmesg 明确显示 phy1-ap0 disable/enable 周期 |
| 🥈 | **旧设备关联耗时长触发芯片切换** | 高 | logread 显示每次新设备连接伴随 odhcpd 路由丢失 |
| 🥉 | **bridge 端口状态变化导致路由抖动** | 中 | br-lan port 5 进入 disabled 状态 |

### 7.3 修复方案

**核心思路**：不删除 5G AP（用户明确需要），而是从三个层面修复：
1. **减少触发**：对齐频宽，减少芯片模式切换
2. **快速恢复**：路由看门狗 + 改进的认证脚本
3. **开机加速**：init.d 启动脚本 + @reboot cron

#### 修复 1：5G AP 频宽对齐（减少触发）

```bash
# 将 5G AP 频宽从 80MHz 降到 40MHz，与 STA 保持一致
uci set wireless.radio1.htmode='HE40'
uci commit wireless
wifi reload
```

> **小白学**：频宽越大 = 速度越快但覆盖越小。80MHz 理论最大 1200Mbps，40MHz 理论最大 573Mbps。对于校园网 100Mbps 出口，40MHz 完全够用。而且 STA 连校园网本来就是 40MHz，AP 设 80MHz 反而会导致芯片需要在两种模式间切换——这本身就是触发故障的根源。

#### 修复 2：路由看门狗脚本（快速恢复）

新建 `/etc/campus_network/route_watchdog.sh`：

```bash
#!/bin/sh
# 每20秒检测默认路由是否丢失
# 如果丢失，立即尝试 DHCP 续租或 WiFi 重连

check_route() {
    ip route show default 2>/dev/null | grep -q "phy1-sta0"
}

check_sta() {
    iw dev phy1-sta0 link 2>/dev/null | grep -q "Connected"
}

main() {
    if check_route; then
        exit 0  # 路由正常，什么都不做
    fi
    
    # 路由丢失！立即恢复
    if check_sta; then
        ifup wwan     # STA还在，只需续租IP
    else
        wifi reload   # STA也断了，需要完整重连
        ifup wwan
    fi
    
    # 恢复后检查是否需要重新认证
    if ! ping -c 2 -W 3 8.8.8.8 > /dev/null 2>&1; then
        /etc/campus_network/auto_login.sh &
    fi
}
```

> **小白学**：
> - `ip route show default` 查默认路由。路由表是 Linux 内核决定"包往哪发"的规则表。如果默认路由没了，内核就不知道如何把包发到外网。
> - `ifup wwan` 触发 udhcpc 重新向校园网 DHCP 服务器请求 IP 和路由。这个过程只需要 2-3 秒。
> - 看门狗和认证脚本是**解耦**的：看门狗只管路由恢复（快），认证脚本只管 Portal 认证（慢）。

#### 修复 3：增强版认证脚本（v2.0）

在 `/etc/campus_network/auto_login.sh` 中新增：

- **`check_route()`**：检测默认路由是否存在
- **`recover_route()`**：路由丢失时自动恢复（DHCP 续租 / WiFi 重连）
- **改进的 `main()`**：先查路由 → 再查网络 → 最后才走完整认证

```bash
main() {
    # 第一步：检查网络连通性
    if check_network; then
        log "网络已连通"
        return 0
    fi
    
    # 第二步：网络不通，先检查路由
    if ! check_route; then
        log "⚠️ 默认路由丢失，尝试恢复..."
        if recover_route; then
            if check_network; then
                log "✅ 路由恢复后网络已连通"
                return 0
            fi
        fi
    fi
    
    # 第三步：完整认证（路由存在但网络不通 = 认证过期）
    full_auth
}
```

> **小白学**：旧版脚本只做一个检测（ping 8.8.8.8），不通就重认证。但"不通"有两种可能：路由丢了（需要续租 IP）和认证过期（需要 Portal 重登）。新版区分了这两种情况，对症下药。

#### 修复 4：开机快速重连（init.d + @reboot）

```bash
# /etc/init.d/campus_auth (START=99, 在网络就绪后执行)
# 等待 WiFi STA 连接（最多 60 秒）
# 等待获取 IP（最多 30 秒）
# 立即触发认证
```

Cron 备份：
```
@reboot sleep 30; /etc/campus_network/auto_login.sh
```

> **小白学**：
> - `START=99` 是 OpenWrt init 系统的优先级。数字越大越晚启动。99 意味着"等所有网络服务都好了再运行我"。
> - `@reboot` 是 cron 的特殊时间——不是定时而是"开机时运行一次"。后面加 `sleep 30` 是给 WiFi 和 DHCP 留出初始化时间。
> - 两种机制同时存在是为了冗余：init.d 是主力（快），@reboot 是兜底（稳）。

#### 修复 5：Cron 优化（业内标准间隔）

```
# 路由看门狗 — 每20秒（业内推荐的快速恢复间隔）
* * * * * /etc/campus_network/route_watchdog.sh
* * * * * sleep 20; /etc/campus_network/route_watchdog.sh
* * * * * sleep 40; /etc/campus_network/route_watchdog.sh

# 校园网认证 — 每3分钟（认证过期检测，不需要太频繁）
*/3 * * * * /etc/campus_network/auto_login.sh

# 开机快速连接
@reboot sleep 30; /etc/campus_network/auto_login.sh
```

> **为什么是 20 秒和 3 分钟？**
> - **路由看门狗 20 秒**：路由丢失是"急症"，需要快速抢救。20 秒是业内高可用系统的标准心跳间隔——足够快到用户几乎无感知（恢复只需 2-3 秒 DHCP + 看门狗最多等 20 秒），又不至于频繁到消耗 CPU。
> - **认证 3 分钟**：Portal 认证有效期通常几小时，过期是"慢病"，3 分钟检查一次绰绰有余。之前 1 分钟太激进，没必要地频繁访问认证服务器。

### 7.6 踩坑记录

#### 坑 6：同频 AP+STA 的隐形陷阱

**现象**：5G WiFi 芯片同时做 AP 和 STA，平时看起来工作正常，但新设备连接时全网卡顿。

**根因**：mt76 驱动在处理 AP 侧客户端关联时，会短暂重配置无线硬件。这导致同一 radio 上的 STA 接口也受影响。频宽不匹配（AP 80MHz vs STA 40MHz）加剧了这个问题——每次模式切换需要更多时间。

**解决**：对齐频宽 + 路由看门狗快速恢复。

**通用技巧**：OpenWrt 上同频 AP+STA（"WiFi 中继"模式）本身就是一个 tradeoff。最佳实践是：
1. 尽量用不同频段（2.4G AP + 5G STA，如本项目）
2. 如果必须同频，确保频宽和信道参数完全一致
3. 始终部署路由/连接监控和自动恢复机制

#### 坑 7：odhcpd "No default route" 的含义

**现象**：日志频繁出现 `odhcpd: No default route present`，但实际上 IPv4 默认路由还在。

**根因**：odhcpd 监控的是 IPv6 默认路由。当 WiFi 接口短暂抖动时，IPv6 路由先消失（因为 IPv6 需要 RA 通告），IPv4 路由可能还在。但 odhcpd 一旦检测到无默认路由，就会把 RA lifetime 设为 0——这会让所有 IPv6 客户端停止使用这个网关。

**教训**：这个日志是"煤矿里的金丝雀"——它不直接导致断网，但它是底层 WiFi/路由问题的早期信号。

### 7.7 教训总结

| 教训 | 说明 |
|------|------|
| 排查问题先 SSH 上去看实际配置 | 不要依赖文档描述，路由器上的配置可能已经被改过 |
| dmesg 比 logread 更能揭示底层问题 | 内核日志记录了驱动级别的状态变化 |
| 同频 AP+STA 是对硬件的极限压榨 | 不是不能用，但必须做好防护和快速恢复 |
| 频宽不匹配是隐形杀手 | 看起来都能工作，实际上在频繁切换 |
| 路由监控比认证监控更重要 | 路由丢了一切都白搭，认证只是"最后一公里" |
| 解耦看门狗和认证逻辑 | 各有各的节奏，互不干扰 |

## Phase 10：旧设备重连断网修复 v2——快速恢复守护进程（2026-06-08）

> **背景**：Phase 7 的修复（频宽对齐 + cron看门狗）缓解了问题，但未根治。
> 用户反馈：旧设备离开后又回来，挂机游戏仍然会长时间断网，提示"网络异常"。

### 10.0 问题复现与深度诊断

#### 用户报告的症状

> "每当我旧设备离开后又回来，我挂机的游戏又会直接断开，而且不是短暂的断开，而是一段比较长时间的断开导致他提示我网络异常"

**关键词**：不是短暂断开、比较长时间、游戏提示网络异常。

这说明恢复链路太慢——TCP 长连接（游戏、视频通话）对中断的容忍度通常只有 10-30 秒，超过就会触发"网络异常"弹窗。

#### 深入 SSH 诊断

```bash
ssh ax3000t

# 发现1：phy1 同时运行 STA 和 AP
iw dev phy1-ap0 info    # channel 52, 40MHz, AP
iw dev phy1-sta0 info   # channel 52, 40MHz, STA
# ⚠️ 同一芯片两个角色！

# 发现2：cron 看门狗每20秒才检测一次
crontab -l
# * * * * * /etc/campus_network/route_watchdog.sh
# * * * * * sleep 20; /etc/campus_network/route_watchdog.sh
# * * * * * sleep 40; /etc/campus_network/route_watchdog.sh

# 发现3：认证日志显示历史断网事件

# [Watchdog] STA 已断开，重新连接 WiFi...
# → 恢复时间通常在 30-50 秒
```

### 10.1 根因分析 v2：为什么断网时间这么长？

Phase 7 已经发现了"同芯片 STA+AP 互相干扰"的根因。但用户反馈问题仍然存在，说明**恢复速度不够快**。

#### 完整恢复链路的时间分解

```
旧设备重连 5G AP
  ↓ (驱动处理 WPA2 握手，挂起 STA 接口)
phy1-sta0 断开 + 默认路由丢失  
  ↓ (等待 cron 看门狗触发 — 最长 20 秒！)      ← 🔴 瓶颈1
route_watchdog.sh 检测到路由丢失
  ↓ (wifi reload 完整重置无线 — 5~8 秒)          ← 🔴 瓶颈2
STA 重新连接到 henu-student
  ↓ (DHCP 获取 IP — 2~3 秒)
ifup wwan 完成
  ↓ (校园网 Portal 认证过期，需重新认证 — 10~15秒) ← 🔴 瓶颈3
恢复外网
────────────────────────────────────────────────
总断网时间: 最长 48 秒，最短 25 秒
游戏 TCP 超时阈值: ~20 秒
→ 必然触发"网络异常" ❌
```

> **小白学**：TCP 长连接有一层"心跳"机制。如果超过一定时间（通常是 15-30秒）没收到对方的数据包，TCP 就认为连接断了，触发"Connection timeout"。游戏就是靠这个来判断网络状态的。

#### 三个瓶颈的量化分析

| 瓶颈 | 当前方案 | 耗时 | 优化方向 |
|------|----------|:---:|------|
| 检测 | cron 轮询 20秒 | **0~20s** | 持续监控 → 1s |
| 恢复 | `wifi reload` 全量重置 | **5~8s** | `iw connect` 直连 → 2s |
| 认证 | 三步 Portal 认证 | **10~15s** | 保持会话不过期 |
| **合计** | | **25~48s** | **目标 < 5s** |

### 10.2 修复方案

**核心思路**：三个层面同时优化——

| 层面 | 策略 | 手段 |
|------|------|------|
| 🛡️ **预防** | 减少 STA 被干扰的可能 | 锁定信道、关闭省电/后台扫描、hostapd 优化 |
| ⚡ **检测** | 从 cron 轮询改为持续监控 | 守护进程每秒检测 |
| 🚀 **恢复** | 从全量 wifi reload 改为精准恢复 | `iw connect` 直连 + DHCP 续租 |

#### 修复 1：无线配置优化（预防层）

```bash
# 1.1 锁定 5G 信道 — 防止 ACS 自动扫描触发 radio reset
uci set wireless.radio1.channel='52'         # 从 auto → 52

# 1.2 hostapd 优化 — 减少客户端关联时的无线中断
uci set wireless.wifinet0.disassoc_low_ack='0'      # 不因低ACK踢客户端
uci set wireless.wifinet0.skip_inactivity_poll='1'  # 跳过非活跃轮询
uci set wireless.wifinet0.wpa_group_rekey='86400'   # 每天才换组密钥

# 1.3 STA 优化 — 关闭省电和后台扫描，防止主动离线
uci set wireless.default_radio1.powersave='0'       # 关闭省电模式
uci set wireless.default_radio1.bgscan='0'          # 禁止后台扫描

uci commit wireless
```

> **小白学**：
> - **ACS (Automatic Channel Selection)** = 自动信道选择。设为 `auto` 时，驱动会定期扫描所有信道找最优的——这个扫描过程中 STA 会短暂离线。
> - **disassoc_low_ack**：hostapd 默认会踢掉 ACK（确认信号）弱的客户端。设为 0 关闭这个行为，减少不必要的重关联。
> - **skip_inactivity_poll**：hostapd 默认会定期轮询每个客户端是否还活着。跳过这个轮询可以减少无线芯片的压力。
> - **powersave**：WiFi 省电模式。当 STA 进入省电模式时，它会告诉 AP "我要睡觉了"，AP 会缓存数据。问题是在同芯片 AP+STA 场景下，STA "睡觉"可能导致芯片进入低功耗状态，影响 AP 侧的客户端感知。关闭省电意味着略微多耗一点电（毫瓦级别，对插电路由器可以忽略）。
> - **bgscan**（Background Scan）：wpa_supplicant 在后台扫描附近 WiFi 信号，用于漫游决策。禁用后 STA 不会主动切换信道扫描。

#### 修复 2：快速恢复守护进程（检测+恢复层）

这是本次修复的**核心**。我们不依赖 cron 每 20 秒轮询，而是写一个**持续运行的守护进程**。

**设计思路**：

```
while true:
    if ping 8.8.8.8 通:
        sleep 1秒
        continue                      ← 网络正常，什么都不做

    if 默认路由丢失:
        if STA 还连着:
            DHCP 续租即可              ← 只需 2-3 秒
        else:
            iw connect 直连 STA        ← 只需 2-3 秒（不用 wifi reload！）
            DHCP 续租

    if 路由存在但网络不通:
        触发认证脚本                   ← 认证过期，需要 Portal 重登

    sleep 1秒
```

**关键优化细节**：

```bash
# 旧方案（慢）：wifi reload → 重置所有无线 → 8秒
wifi reload 2>/dev/null
sleep 5

# 新方案（快）：直接告诉 STA "连这个 SSID，用这个频率"
#            跳过扫描阶段 → 2秒
iw dev phy1-sta0 disconnect 2>/dev/null
sleep 0.5
iw dev phy1-sta0 connect "henu-student" 5260
```

> **小白学**：
> - `wifi reload` 的代价：它会停止 radio0 和 radio1 上所有虚拟接口（包括 2.4G AP！），然后重新创建它们。这意味着不仅是 STA 断了，宿舍的 2.4G WiFi 也短暂消失。总耗时 5-8秒。
> - `iw connect` 只操作 phy1-sta0 这一个接口，不影响 2.4G AP。而且我们直接指定了目标频率（5260 MHz = 信道 52），让驱动跳过信道扫描。耗时 ~2 秒。
> - **5260 MHz 是什么？** WiFi 信道 52 的中心频率。信道编号是给人看的，频率是给机器用的。`iw connect SSID FREQ` 直接告诉驱动"去这个地方找这个 WiFi"，省掉了扫描步骤。

**守护进程的层级恢复策略**：

```
层级1 (1秒检测): ping 8.8.8.8 通了 → 正常
层级2 (3秒恢复): 路由丢了 + STA在 → DHCP续租
层级3 (5秒恢复): 路由丢了 + STA不在 → iw connect直连 + DHCP
层级4 (60秒冷却): 路由在但网络不通 → 触发认证（不频繁重试）
层级5 (30秒兜底): 持续失败 → wifi reload 完整重置
```

> **为什么需要"冷却时间"？** 如果认证服务器暂时故障或者校园网 DHCP 抽风，不加冷却会每秒钟重试一次认证，对校园网服务器造成压力（可能被当作攻击封掉你的 IP）。60 秒冷却意味着两次认证之间至少隔 1 分钟。

#### 修复 3：替换 cron 看门狗

旧的 cron 看门狗：
```
* * * * * /etc/campus_network/route_watchdog.sh          ← 每分钟第0秒
* * * * * sleep 20; /etc/campus_network/route_watchdog.sh ← 第20秒
* * * * * sleep 40; /etc/campus_network/route_watchdog.sh ← 第40秒
```

新的守护进程：
```bash

/etc/init.d/fast_recovery enable
/etc/init.d/fast_recovery start
```

cron 只保留认证检测（每 3 分钟，不需要频繁）：
```
*/3 * * * * /etc/campus_network/auto_login.sh
@reboot sleep 30; /etc/campus_network/auto_login.sh
0 4 * * 1,4 /sbin/reboot
```

> **为什么还要保留 3 分钟的认证检测？** 守护进程虽然快，但它的认证触发有 60 秒冷却。如果守护进程在这一分钟内连续遇到了需要认证的情况，冷却时间内不会重试。cron 的 3 分钟认证检测作为兜底，确保认证永远不会漏掉。

### 10.4 关键洞察

#### 为什么"旧设备"更容易触发问题？

诊断时发现两台正常设备（LAPTOP-TAYLOR 和 Honor 400）和一台 OPPO Reno 14 Pro 的详细信息：

| 设备 | 天线 | 关联耗时 | 对芯片的冲击 |
|------|:---:|:---:|:---:|
| Honor 400 | 2×2 | ~50ms | 极小 |
| LAPTOP-TAYLOR | 2×2 | ~50ms | 极小 |
| OPPO Reno 14 Pro | **1×1** | **200~500ms** | 较大 |

OPPO 的 `long preamble`（长前导码）意味着它的 WiFi 固件使用较旧的握手方式——关联帧交换更多、WPA 握手更慢。整个过程耗时是正常设备的 4-10 倍。

> **前导码（Preamble）是什么？** WiFi 帧的最开头部分，用于接收端同步时钟和检测帧的起始位置。**Short preamble** 只有 96 微秒（72 bits），**Long preamble** 需要 192 微秒（144 bits）。长期来看，long preamble 会让每个帧的开销多出一倍——在芯片需要频繁切换 STA/AP 模式的情况下，这些微秒级的开销累积起来就变成了毫秒级的延迟。

**正常设备**：关联耗时 50ms → STA 断连窗口窄 → 往往不出问题。
**"旧设备"**：关联耗时 200-500ms → STA 断连窗口宽 → **几乎每次都会触发路由丢失**。

这就是为什么用户感觉"只有旧设备回来才断网"——不是因为"旧"，而是因为设备硬件差异导致了不同的关联耗时。

#### 单天线 vs 双天线的实际影响

OPPO Reno 14 Pro 只有 1 根天线（NSS=1）：
```
2×2 设备理论最大速率: 573.5 Mbps (40MHz WiFi 6)
1×1 设备理论最大速率: 286.7 Mbps (40MHz WiFi 6)  ← 减半
```

但 286Mbps 对于校园网 100Mbps 出口完全够用。用户体验差异主要来自关联行为（long preamble）而非速率。

### 10.6 踩坑记录

#### 坑 8：pidof 无法检测 shell 守护进程

**现象**：启动守护进程后，`pidof fast_recovery_daemon.sh` 返回空，init.d 脚本认为启动失败。

**根因**：`pidof` 匹配的是**可执行文件名**。procd 以 `/bin/sh /etc/campus_network/fast_recovery_daemon.sh` 启动守护进程，进程的可执行文件是 `/bin/sh`，不是 `fast_recovery_daemon.sh`。

**解决**：用 `ps | grep fast_recovery` 检测进程是否运行，不用 `pidof`。

#### 坑 9：wifi reload 后的抖动

**现象**：wifi reload 后的 5-10 秒内，路由会经历"恢复→再丢→再恢复"的抖动。

**根因**：wifi reload 触发 radio 重建 → STA 重连 → DHCP 获取 IP → 设置路由。但在路由设置完成前，内核可能会短暂看到一个没有路由的中间状态。udhcpc 的 lease renew 过程也可能多次更新路由表。

**解决**：守护进程在恢复成功后加 1 秒检查间隔，避免在 DHCP 协商过程中反复触发恢复。这不是 bug，而是过渡态的正常行为。

### 10.7 教训总结

| 教训 | 说明 |
|------|------|
| cron 不适合做快速恢复 | 最小粒度是分钟，20秒的多 cron hack 本质上是对机制缺陷的补偿 |
| 持续守护进程 > 轮询 | 1 秒 vs 20 秒，这是 20 倍的响应速度差异 |
| `iw connect SSID FREQ` 比 `wifi reload` 快 3-4 倍 | 精准操作 vs 全量重置，且不影响其他接口 |
| 预防 > 恢复 | 锁定信道、关闭省电/扫描是最便宜的"修复" |
| 关注关联行为而非速率 | 1×1 设备速率减半不是问题，long preamble 的频繁重关联才是 |
| 检测网络故障需要分层 | ping 通 → 正常 / 路由丢 → DHCP / STA 断 → 重连 / 路由在但 ping 不通 → 认证 |
