## 项目概述

### 背景
河南大学校园网使用 **henu-student** WiFi + Portal 认证。路由器需要：
1. 用 WiFi（5G STA 模式）连接校园网
2. 自动完成三步 Portal 认证（获取参数 → 验证身份 → 门户登录）
3. 持续监控网络状态，断网自动恢复
4. 作为宿舍 LAN 设备的网关，提供 NAT 上网

### 核心架构

```
                   校园网                              宿舍内网
            ┌──────────────┐                 ┌──────────────────┐
            │  henu-student │  5G STA(WAN)    │   AX3000T 路由器  │
            │   (WiFi AP)   │◄──────────────►│  phy1-sta0 (WAN)  │
            │   校园网交换机  │                 │  br-lan (LAN)     │
            │   认证服务器    │                 │  phy0-ap0 (2.4G)  │
            │  172.29.35.x  │                 │  phy1-ap0 (5G)    │
            └──────────────┘                 └────────┬─────────┘
                                                      │
                                              ┌───────┼───────┐
                                              │       │       │
                                            手机    笔记本   台式机
```

## 步骤0：刷机（开箱到 ImmortalWrt）

### 0.1 原厂状态

小米原厂固件基于旧版 OpenWrt 但**锁死了 SSH**：

| 能力 | 原厂固件 | 刷机后 |
|------|:--------:|:------:|
| SSH 远程登录 | ❌ | ✅ |
| 安装任意软件 | ❌ | ✅ |
| 写脚本自动化 | ❌ | ✅ |
| WiFi 做客户端 | ❌ | ✅ (STA 模式) |
| 存储可用 | ~10MB | 78MB |
| 小米后台 | ✅ | ❌ |

### 0.2 刷机流程（三步）

```
开箱初始化 → 解锁 SSH → 备份 Factory 分区 → 刷入大分区 U-Boot → 刷 ImmortalWrt
```

#### ① 解锁 SSH
使用 **[XMiR-Patcher](https://github.com/openwrt-xiaomi/xmir-patcher)**（Python 工具）：
- 通过小米固件漏洞获取 SSH 访问
- 备份 Factory 分区（**必须备份！** 含 WiFi 校准数据，丢了信号变差）

#### ② 刷入大分区 U-Boot
- 扩大 flash 分区表，给 overlay 腾出 ~70MB 空间
- **注意**：不要动 mtd0-mtd4（U-Boot 和 BL2），否则变砖

#### ③ sysupgrade 刷 ImmortalWrt
```bash
# 通过 SSH 上传固件
scp immortalwrt-24.10.5-...-ax3000t-squashfs-sysupgrade.bin root@192.168.31.1:/tmp/

# 刷写
sysupgrade -n /tmp/*.bin
# 等待约 2-3 分钟自动重启
```

### 0.3 固化（重启不丢配置）

刷机后 `/overlay` 存在于 Flash 上（非内存盘），所以重启后配置保留。

```bash
# 验证固化
df -h /overlay
# → Size: 78M (大分区成功)

mount | grep overlay
# → overlayfs:/overlay on / type overlay
```

### 0.4 刷机前后 IP 变化

| 项目 | 原厂固件 | ImmortalWrt |
|------|----------|-------------|
| 默认 IP | 192.168.31.1 | 192.168.1.1 |
| 默认密码 | 无 | root 无密码 |
| 管理界面 | miwifi.com | http://192.168.1.1 |

## Phase 0：前置核心概念

### 概念1：SSH（远程登录路由器）

```bash
ssh root@192.168.1.1
```

相当于远程桌面，只不过用键盘控制。配置免密后：
```bash
# 配置 ~/.ssh/config
Host ax3000t
    HostName 192.168.1.1
    User root
    IdentityFile ~/.ssh/ax3000t_rsa

# 之后只需：
ssh ax3000t
```

### 概念2：OverlayFS（修改持久化原理）

OpenWrt 使用 OverlayFS 叠加文件系统：

```
/rom (squashfs, 只读)           ← 出厂固件，不可修改
   └── 上层叠加 /overlay (ubifs, 读写)  ← 你的所有修改
           └── 你看到的是合并后的 /       ← 完整系统
```

- **重启 → 不丢失**（因为 /overlay 在 Flash 上）
- **恢复出厂** = 清空 `/overlay`
- 好处：系统分区坏了也能恢复（/rom 永远不会被改坏）

### 概念3：UCI（统一配置接口）

```
uci set wireless.radio0.channel='52'     # 设置
uci commit wireless                       # 保存
wifi reload                               # 生效
```

OpenWrt 用 UCI 统一管理所有配置（/etc/config/ 下的文件）。相当于 Windows 注册表。

常见配置集：
| 文件 | 管理内容 | 相关命令 |
|------|----------|---------|
| `/etc/config/network` | 网络接口、路由 | `uci show network` |
| `/etc/config/wireless` | WiFi 配置 | `uci show wireless` |
| `/etc/config/dhcp` | DHCP、DNS | `uci show dhcp` |
| `/etc/config/firewall` | 防火墙规则 | `uci show firewall` |

### 概念4：WiFi 的三种模式

| 模式 | 作用 | 类比 |
|------|------|------|
| **AP** (Access Point) | 发射 WiFi，设备连接它 | 普通路由器 |
| **STA** (Station) | 连接其他 WiFi 热点 | 手机的 WiFi 连接 |
| **AP+STA** | 同时发射 + 连接 | 中继器/无线 WAN |

本项目使用的是最难的 AP+STA 双模——**同一个 5G 无线电同时连校园网 + 给宿舍设备上网**。

### 概念5：NAT（网络地址转换）

- 内网设备：`192.168.1.x`
- 路由器 WAN：`10.36.x.x`（校园网）
- NAT 把 `192.168.1.x:端口` ⇄ `10.36.x.x:其他端口`
- **MASQUERADE**：特殊的 SNAT，WAN IP 变化时自动适配

### 概念6：DHCP 租约

设备连上路由器时获得一个内网 IP，这个分配有有效期（租约）：
- **12h**：默认值，设备半天不回家，重新连需要完整 DHCP 协商
- **168h（7天）**：本项目的优化值，设备一周内回来用快速续租
- 租约过期后的重连会产生大量广播帧，可能影响到其他设备

### 概念7：守护进程 vs Cron

| | Cron | 守护进程 |
|------|------|---------|
| 精度 | 分钟级 | 秒级 |
| 资源 | 每次启动新进程 | 常驻内存 |
| 适合 | 低频周期性任务 | 高频率实时监控 |
| 本项目 | 每5分钟检测认证 | 每秒检测路由状态 |

## Phase 3：WiFi WAN 配置

这是最核心的技术挑战——**用路由器 5G 无线电连接校园网**。

### 思路

```
原厂：phy0=2.4G AP，phy1=5G AP（两个都是 AP）
我们需要：phy0=2.4G AP，phy1=5G AP+STA（5G 双模）
```

### 操作

```bash
# ① 创建 wwan 接口（WAN 侧）
uci set network.wwan=interface
uci set network.wwan.proto='dhcp'
uci set network.wwan.device='phy1-sta0'
uci commit network

# ② 配置 STA（连接校园网）
uci set wireless.radio1.channel='52'        # 锁定信道
uci set wireless.radio1.htmode='HE40'       # 40MHz频宽
uci set wireless.radio1.band='5g'

# ③ 创建 STA 虚拟接口
uci set wireless.sta0=wifi-iface
uci set wireless.sta0.device='radio1'
uci set wireless.sta0.mode='sta'
uci set wireless.sta0.network='wwan'
uci set wireless.sta0.ssid='henu-student'
uci set wireless.sta0.encryption='none'

uci commit wireless
wifi reload
```

### 关键配置细节

| 参数 | 值 | 原因 |
|------|-----|------|
| radio1.channel | 52 | 锁定 UNII-2 信道，避免 ACS 周期断网 |
| radio1.htmode | HE40 | 40MHz 减少干扰，与 STA 对齐 |
| radio1.country | CN | 遵守中国法规，启用合法信道 |

## Phase 4：校园网认证脚本

### 三步认证流程

```
第一步：获取授权码
    POST → 172.29.35.27:8088/aaa-auth/api/v1/auth
    body: campusCode + username + password + operatorSuffix
    response: {"code":1, "data":"user@henuyd"}     ← 成功后拿到授权

第二步：验证身份
    POST → 172.29.35.27:8882/user/check-only
    body: username + password + operatorSuffix
    response: {"code":1, "msg":"SUCCESS"}           ← 身份验证通过

第三步：Portal 登录
    GET → 172.29.35.36:6060/quickauth.do
    params: userid + passwd + wlanuserip + ...（加密参数）
    response: {"message":"认证成功"}                 ← 正式上线
```

### 脚本架构（v3.0）

```
auto_login.sh
├── preflight_check()     ← 认证前检查（IP、路由、服务器可达）
├── first_auth()          ← 第一步 + 重试机制（最多3次）
├── second_auth()         ← 第二步
├── portal_auth()         ← 第三步（带加密参数）
├── check_network()       ← 认证后验证（ping 8.8.8.8）
├── recover_route()       ← 路由丢失时自动恢复
└── main()                ← 主流程（检测→恢复→认证）
```

### 核心认证函数
```bash
#!/bin/sh
USERNAME="你的学号"
PASSWORD="校园网密码"
OPERATOR_SUFFIX="@henuyd"    # @henuyd=移动 @henujy=教育网
CAMPUS_CODE="07cdfd..."      # 校验证码

first_auth() {
    RESPONSE1=$(wget -q -O - --timeout=10 \
        --post-data="campusCode=${CAMPUS_CODE}&username=${USERNAME}&password=${PASSWORD}&operatorSuffix=${OPERATOR_SUFFIX}" \
        --header="Content-Type: application/x-www-form-urlencoded" \
        --header="Referer: http://172.29.35.36:6060/" \
        "http://172.29.35.27:8088/aaa-auth/api/v1/auth")
    
    if echo "$RESPONSE1" | grep -q '"code":1'; then
        return 0  # 成功
    fi
    return 1  # 失败重试
}
```

## Phase 6：CAKE 智能流控

### 原理

**CAKE**（Common Applications Kept Enhanced）是现代化的 QoS 队列算法：
- 带宽公平分配（多人下载不互相抢）
- 低延迟优先（游戏/视频通话不卡）
- 内置流量整形（不需要另一套限速）

### 配置
```bash
tc qdisc replace dev ifb4wwan root cake \
    bandwidth 110mbit \          # 校园网下行带宽
    diffserv4 \                  # 4级差分服务
    nat \                        # 识别 NAT 后的真实源 IP
    wash \                       # 清除 DSCP 标记
    cake overrides \             # 覆盖客户端 DSCP
    ingress                      # 入站整形

tc qdisc replace dev wwan root cake \
    bandwidth 20mbit \           # 上行为下行约1/5
    diffserv4 nat wash
```

### 效果
| 场景 | 无 CAKE | 有 CAKE |
|------|:-------:|:-------:|
| 一人下载 + 一人打游戏 | 游戏延迟 200ms+ | 游戏延迟 <30ms |
| 三人同时看视频 | 轮流缓冲 | 同时流畅 |
| 宿舍多人同时用 | 体验差 | 体验稳定 |

## Phase 8-10：设备重连断网修复

这是项目中最复杂的问题，经历了 **3 次迭代**。

### 问题现象
已有设备连着的 WiFi，另一个设备离开后回来重连 → 所有设备短暂断网。

### 修复历程

#### v1（Phase 8）：频宽对齐 + 看门狗

| 修复 | 效果 |
|------|------|
| 5G AP 频宽 80MHz→40MHz | 减少 radio reset |
| 锁定 ch52 | 消除 ACS 周期断连 |
| 路由看门狗 cron 20秒 | 丢路由后 20s 恢复 |
| 问题 | 20 秒粒度对挂机游戏不够 |

#### v2（Phase 10）：快速恢复守护进程

用 Shell 守护进程替代 cron，**秒级恢复**：

```bash
# 核心循环
while true; do
    if ! ping -c 1 -W 1 8.8.8.8 >/dev/null 2>&1; then
        recover_route  # 恢复路由
        auto_login     # 重新认证
    fi
    sleep 1
done
```

使用 procd 管理守护进程：
```bash
# /etc/init.d/fast_recovery
start_service() {
    procd_open_instance
    procd_set_param command /etc/campus_network/fast_recovery_daemon.sh
    procd_set_param respawn 3600 5 0  # 崩溃后1小时重试，最多5次
    procd_close_instance
}
```

| 指标 | v1（cron） | v2（守护进程） |
|------|:---------:|:------------:|
| 检测间隔 | 20 秒 | 1 秒 |
| 恢复时间 | 最多 20 秒 | 1-3 秒 |
| 资源占用 | 低 | 极低（sleep 1） |

## Phase 9：校园网 DNS 修复

### 问题
校园网 DHCP 提供两个 DNS：
| DNS | 角色 | 问题 |
|-----|------|------|
| 111.6.174.198 | 河南移动 | 对部分 henu 域返回内网 IP |
| 114.114.114.114 | 公共 DNS | 不知道内网 IP |

### 四层修复方案

```
第一层：/etc/hosts（最可靠）
  zwyy.henu.edu.cn       → 202.196.96.29  (111DNS返回125.x❌)
  software.henu.edu.cn   → 58.212.123.41   (111DNS解析为空❌)
  net.henu.edu.cn        → 172.31.7.4      (公网IP校园内不可达)
  superhuazai.me         → 104.21.18.208   (间歇解析异常)

第二层：UCI 精确转发
  xg.henu.edu.cn  → 114.114.114.114  (111返回172.31.0.6❌)
  jwgl.henu.edu.cn → 111.6.174.198    (114返回211.x❌)

第三层：dnsmasq rebind 豁免
  rebind-domain-ok=/henu.edu.cn/
  rebind-domain-ok=/ntp.org.cn/

第四层：过滤 IPv6
  filter_aaaa=1  (路由器无 IPv6 路由)
```

一键修复：`sh /etc/campus_network/dns_fix.sh`

## Phase 12：启动认证循环修复

### 问题
路由重启后，守护进程立即触发认证，但 STA 还没连上校园网 → 认证失败 → 循环重试。

### 修复

```bash
# v2.0 守护进程加入启动等待期
startup_wait() {
    local max_wait=30
    local waited=0
    while [ $waited -lt $max_wait ]; do
        if iw dev phy1-sta0 link 2>/dev/null | grep -q "Connected"; then
            break  # STA 已就绪
        fi
        sleep 1
        waited=$((waited + 1))
    done
}

# v3.0 认证脚本加入预检查
preflight_check() {
    # 1. STA 有 IP？
    ip addr show phy1-sta0 | grep -q "inet " || return 1
    # 2. 有默认路由？
    ip route show default | grep -q "phy1-sta0" || return 1
    # 3. 认证服务器可达？
    ping -c 1 -W 2 172.29.35.27 >/dev/null 2>&1 || return 1
    return 0
}
```

**效果**：启动恢复时间从 8 分钟降到 3 分钟。

## Phase 13：断联修复 v3

### 问题复现
设备 A 离开半天（超过 12h）后回来重连 WiFi → 其他设备断网。短时间离开则不会。

### 排查发现
| 检查项 | 2.4G | 5G | 问题 |
|--------|:----:|:--:|------|
| disassoc_low_ack | **1**（踢弱信号） | 0 | ⚠️ 不一致 |
| skip_inactivity_poll | **0**（轮询） | 1 | ⚠️ 不一致 |
| DHCP 租约 | **12h** | — | ⚠️ 精准匹配"半天" |
| UAPSD | 开启 | 开启 | ⚠️ 唤醒延迟 |
| PMKSA 缓存 | 关闭 | 关闭 | ⚠️ 重连握手慢 |

### 根因链

```
DHCP 租约 12h 过期
    │
    ▼
设备回家重连 → DHCP DISCOVER（广播，非快速 REQUEST）
    │
    ▼
大量广播帧（DHCP + ARP + EAPOL 四次握手）
    │
    ├─→ 2.4G: disassoc_low_ack=1 → 弱信号设备 ACK 来不及 → 被踢下线
    ├─→ 5G: 广播帧干扰 WED 硬件加速 → 其他设备短暂异常
    └─→ UAPSD 省电模式 → 唤醒延迟 → 更容易错过 ACK
```

### 修复（5项）

```bash
# 1. 2.4G 配置对齐 5G
uci set wireless.default_radio0.disassoc_low_ack='0'
uci set wireless.default_radio0.skip_inactivity_poll='1'
uci set wireless.default_radio0.uapsd='0'
uci set wireless.default_radio0.auth_cache='1'

# 2. 5G 也关闭 UAPSD 并开启 PMKSA
uci set wireless.default_radio1.uapsd='0'
uci set wireless.default_radio1.auth_cache='1'
uci set wireless.wifinet0.uapsd='0'
uci set wireless.wifinet0.auth_cache='1'

# 3. DHCP 租约 12h → 168h
uci set dhcp.lan.leasetime='168h'

uci commit wireless && uci commit dhcp
wifi reload
```

### 修复后效果
| 指标 | 修复前 | 修复后 |
|------|:-----:|:------:|
| DHCP 租约 | 12h | 168h ✅ |
| disassoc_low_ack | 2.4G=1, 5G=0 | **对齐=0** ✅ |
| UAPSD | 开启 | **关闭** ✅ |
| PMKSA 缓存 | 关闭 | **开启** ✅ |
| 设备重连断联 | ⚠️ 频繁 | ✅ 已消除 |

## 附录：核心原理解析

### A. OpenWrt 文件系统

```
/
├── /rom/           ← 出厂固件（squashfs，只读）
├── /overlay/       ← 修改存储（ubifs，读写）
├── /bin/           ← 基础命令（busybox 符号链接）
├── /etc/           ← 配置文件
│   ├── /config/    ← UCI 配置文件
│   │   ├── network   ← 网络接口
│   │   ├── wireless  ← WiFi 配置
│   │   ├── dhcp      ← DHCP/DNS
│   │   ├── firewall  ← 防火墙规则
│   │   └── system    ← 系统设置
│   └── init.d/     ← 开机启动脚本
├── /tmp/           ← 临时文件（重启清空）
├── /www/           ← LuCI Web 界面
└── /proc/          ← 内核虚拟文件系统
```

### B. 路由器启动过程

```
上电 → BL2（一级引导）→ U-Boot（二级引导）→ 内核 → init → procd → 启动脚本
  mtd1     mtd4           mtd8(内核+kernel)       ↓
                                            /etc/rc.d/S*按序
                                            ├── boot() 内核模块+文件系统
                                            ├── preinit() 网络就绪
                                            └── init完成 → SSHD+Luci+DNSMASQ
```

### C. Watchdog（看门狗）

```
软件看门狗：系统卡死 → 内核自动重启（/dev/watchdog）
路由看门狗：路由丢失 → 自动修复路由表
硬件看门狗：CPU 内置定时器，需要定期"喂狗"
```

### D. CAKE qdisc 工作原理

```
应用层
  │  Socket Buffer (skb)
  ▼
TC (Traffic Control)
  ├─→ 入站(ingress) → ifb4wwan(重定向) → cake(入站整形)
  └─→ 出站(egress) → cake(出站整形) → 网卡驱动
                       │
                ┌──────┴──────┐
            DiffServ4 分类
          ├─ BK (批量)
          ├─ BE (尽力而为)  ← 默认
          ├─ VI (视频)
          └─ VO (语音)     ← 最高优先级
```

## FAQ：常见问题

### Q1：路由器需要每天重启吗？
不需要。OpenWrt 设计上就是长期运行的。本项目配置了**每周一+周四凌晨4点**自动重启：
- `0 4 * * 1,4 /sbin/reboot`
- 目的：清理 conntrack 表 + 内存碎片
- ⚠️ 不建议每天重启（每次重启会触发 ACS 信道扫描）

### Q2：一个设备重连，为什么其他设备会断网？
见 Phase 13 的根因链。核心：
1. DHCP 12h 租约过期 → 完整 DISCOVER（广播）
2. 广播帧 → 2.4G 弱信号设备 ACK 不及 → 被踢
3. **修复**：DHCP 168h + 关闭 UAPSD + 开启 PMKSA 缓存

### Q3：校园网网站打不开怎么办？
```bash
sh /etc/campus_network/dns_fix.sh
```
四层修复：hosts → UCI 转发 → dnsmasq rebind → IPv6 过滤。

### Q4：如何手动更新认证凭据？
```bash
# SSH 进路由器
ssh ax3000t
vi /etc/campus_network/auto_login.sh
# 修改 USERNAME、PASSWORD、OPERATOR_SUFFIX 三行
# 保存后手动测试
/etc/campus_network/auto_login.sh
```

### Q5：路由器变砖了怎么办？
- 如果 U-Boot 还在 → 按住 Reset 通电 → TFTP 刷入固件
- 如果 BL2 坏了 → 需要 TTL 串口线 + 编程器

### Q6：如何查看当前运行状态？
```bash
# 外网连通性
ping 8.8.8.8

# WiFi 连接状态
iw dev phy1-sta0 link

# 认证日志
tail -20 /tmp/campus_network.log

# 守护进程状态
ps | grep fast_recovery

# 系统运行时间
uptime

# 在线设备
cat /tmp/dhcp.leases
```

### Q7：怎么知道每个设备用了多少流量？
LuCI → Services → Bandwidth Monitor（nlbwmon），显示每台设备的上下行流量。

### Q8：怎么通过 VPN 远程管理家里的路由器？
路由器作为 WireGuard 客户端连接 VPN 服务器，你也连接同一个 VPN 服务器。
然后你的电脑可以直接访问路由器的 LAN IP (`192.168.1.1`)。
```bash
# 查看 VPN 状态
wg show
```
